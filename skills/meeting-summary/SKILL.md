# 会议总结助手

将冗长的会议文字稿转化为结构清晰、结论先行的会议总结。

## 核心原则

会议总结不是流水账。一份好的会议总结，读者花 3 分钟就能抓住全部要点——做了什么决定、为什么这么决定、谁要做什么。

四个关键信念驱动这个 skill 的设计：

- **结论先行**：读者最需要的信息放在最前面。先说"决定了什么"，再说"为什么这么决定"。没人想在一堆讨论细节里翻找结论。
- **捕捉思路而非记录发言**：会议的价值不在于谁说了什么话，而在于哪些想法碰撞出了什么结果。好的总结能让没参会的人理解决策背后的逻辑链。
- **提炼而非堆砌**：案例、类比、轶事是会议思路的一部分，重要的要保留——但必须压缩成结论性的表述，不能原样复述。比如一个精准的类比用一句话嵌入结论里就够了，不需要展开成一个段落讲完整故事。判断标准：这个例子删掉后，读者还能理解决策逻辑吗？能删就删，不能删就压到一句话。
- **忠于事实**：所有内容必须有文字稿依据。可以提炼、可以重组、可以用更精准的语言表达——但不能添加文字稿中没有的信息和观点。当你用自己的语言做了概括性提炼（而非直接引用文字稿的表述），用括号标注一下，让读者能区分"原文说的"和"总结者的提炼"。

## 处理流程

### 第一步：通读与提取

仔细阅读完整的会议文字稿，提取以下信息：

- 会议主题和背景
- 参会人员（如果文字稿中能识别出来）
- 时间信息（如果有）
- 讨论的议题有哪些
- 每个议题下达成了哪些结论、共识
- 关键分歧和不同观点
- 明确的行动项（谁、做什么、什么时候）
- 悬而未决的问题

### 第二步：组织与重构

不要按照会议的时间顺序组织内容。按照**议题**和**重要性**重新组织：

- 将零散的讨论归拢到对应的议题下
- 同一个议题在会议中可能被多次提及（中间岔开聊了别的又回来），要合并整理
- 把最重要的结论和决策提到最前面

### 第三步：撰写总结

按照下方的输出模板撰写。注意语言风格要求。

## 输出模板

```markdown
# [会议主题]

> [一句话概括这次会议最核心的成果或方向]

**时间**：[日期，如有]
**参会人**：[人员列表，如有]

## 核心结论

[这是整份总结中最重要的部分。把所有关键决策和结论浓缩在这里，每条结论用一句话说清楚。读者只看这个部分就能知道会议产出了什么。]

- 结论一的一句话描述
- 结论二的一句话描述
- ...

## 关键讨论

[按议题分组，不按时间排列。优先用列表组织，避免大段叙事。]

### [议题一名称]

[一句话结论或方向]

- [推动决策的关键逻辑点一]
- [关键逻辑点二，如有重要分歧在此体现]
- [重要的类比或案例，压缩成一句话嵌在对应逻辑点里]
- ...

[每条列表项应承载一个独立的逻辑点或决策依据。如果一条列表项需要写两三行才能说清楚，它可能该拆成两条；如果它只是在重复上一条的意思换个说法，该合并或删除。]

### [议题二名称]

...

## 行动项

[明确的待办事项，需要包含责任人和时间节点（如果文字稿中有提及）。]

- [责任人]：[具体任务]（[时间节点，如有]）
- ...

## 未决问题

[会议中提出但未解决的问题，以及可能的后续跟进方向。如果没有未决问题，省略此部分。]

- [问题描述]
- ...
```

如果文字稿中缺少某些信息（比如没有时间、无法识别参会人），直接省略对应字段，不要编造。

如果某个部分没有内容（比如没有明确的行动项），省略该部分。不要为了结构完整而写空洞的内容。

## 篇幅控制

不设硬性行数上限——一个 2.5 小时的深度讨论和一个 30 分钟的站会，信息量本来就不同。但要警惕密度下降：

- 写完后逐条回看列表项，问自己：这条删掉后读者还能理解决策逻辑吗？能删就删
- 如果同一个议题超过了 8~10 条列表项，大概率是把讨论过程当成了逻辑点——重新审视哪些是推动结论的，哪些只是过程中的插曲
- 叙事段落（非列表）单段不超过 3-4 行。超过就考虑拆成列表或压缩措辞

## 语言风格

风格上追求直接、不绕弯子，但语气要松弛。会议总结的读者通常是忙碌的同事，他们想快速了解"发生了什么"，不想读一份措辞紧绷的公文，也不想读一篇有压迫感的文章。

写的时候想象你在跟一个没参会的同事口头转述——清楚、轻松、抓重点。

**要做到**：
- 用短段落和短句，留出呼吸空间——每段不超过 3-4 行
- 直接说本质，不绕弯子。"决定采用方案 A"比"经过充分讨论后大家一致认为方案 A 是更优的选择"好得多
- 用"——"来补充解释或制造停顿
- 用 *斜体* 标注文字稿中特别值得注意的原话或洞见
- 敢于提炼——把 20 分钟的讨论浓缩成 3 句话，只要没丢失关键信息。写完一个议题后问自己：这里面有没有哪句话删掉也不影响读者理解结论？有就删
- 适当保留一点口语化的轻松感。不用每句话都像在写报告，偶尔可以用"说白了"、"简单说"之类的连接词让行文更自然
- 概括性提炼用括号标注，和事实陈述区分开（见核心原则中的说明）

**要避免**：
- 流水账式记录（"然后 A 说了……然后 B 回应说……"）
- 官腔和套话（"与会人员进行了深入的交流和讨论"）
- 过度修饰和冗余表达
- 过度紧绷——不需要每句话都锋利、每段都像在下结论，该轻描淡写的地方轻描淡写
- 添加文字稿中没有的主观评价或推断

**格式规则**：
- 用无序列表和有序列表组织信息，不用表格
- 列表项和引用块（>）开头不要加粗（写"结论描述"而不是"**结论描述**"）
- 粗体仅用于极少数需要强调的关键词，不要滥用
- 适量用 emoji 辅助扫读（标题、结论、行动项等），不要每行都加，也不要堆砌装饰性
- 保持 markdown 格式简洁干净

## 特殊情况处理

- **文字稿质量差**（口语化严重、有大量语气词、前后矛盾）：尽力提炼，对于实在无法判断含义的内容，在总结中标注为存疑
- **说话人识别错误**：录音转文本工具经常把说话人标错（比如多人发言归到同一个人名下，或标注为"说话人 1""说话人 2"等泛称）。需要根据上下文智能纠错——通过发言内容中的自称/他称（"我跟 A 说""B 刚才提到"）、职能线索、对话逻辑（被回应的人不是提问的人）来推断实际说话人。总结中归属观点时以能确认的为准，无法确认的不强行归属。
- **头脑风暴类会议**（没有明确结论）：把"核心结论"改为"核心方向"或"初步共识"，在关键讨论中记录主要思路
- **多主题会议**：按议题重要性排序，而不是按讨论顺序
- **有明显冲突的会议**：客观呈现各方立场和论据，不偏向任何一方
