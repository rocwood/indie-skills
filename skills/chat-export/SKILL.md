# Chat Export

将当前 AI 对话导出为 Markdown 文件。通过整理 LLM 上下文窗口中的内容实现，不依赖平台存储格式，适用于支持对话上下文的 Agent 工具。

**已知限制**：若会话经历上下文压缩或超出窗口，早期内容可能丢失。长会话建议中途分段导出。

## 步骤

### 1. 确定输出路径

按优先级选择，**不要反问用户**，直接决定：

1. 用户明确指定的路径
2. 从工作区上下文推断合理位置（如笔记库的日记目录、项目的 docs 目录等）
3. 工作区根目录

文件名：`YYYYMMDD-{主题关键词}.md`，主题从对话内容自动提炼（2-10 个字，仅使用中文、英文、数字和连字符）。
如果工作区有明确的文件命名惯例，遵循该惯例。

### 2. 回顾上下文，提取对话

扫描上下文窗口，按时间顺序逐轮提取。**不包含"执行导出"这一轮本身。**

**过滤**（只过滤机械性噪音）：

- 系统注入内容（打开的文件列表、附加文件全文、workspace rules、skills、用户环境元数据等）
- 工具调用的参数细节和原始返回值
- AI 内部过程（thinking / chain-of-thought、TODO 状态更新操作等，但 TODO 列表内容本身若体现了有价值的任务拆解则保留）
- 工具动作本身用一句话带过（如"读取了文件 Y"、"执行命令得到结果 Z"），但 AI 基于工具结果的分析和推导属于回复内容，不是工具噪音

**保留规则**：

- 用户消息原样保留。如果用户消息通过 `@文件名` 引用了文件，在消息后用括号注明实际路径，如 `（引用了 path/to/file.md）`
- AI 回复按原始论述复现，包括分析过程、推导逻辑、举例论证、对比论点——不只是结论。如果 AI 原文用了 10 行来分析一个问题，导出也应该接近 10 行，不要压缩成 2 行结论

**兜底原则**：拿不准是否该保留时，保留。信息丢失的代价大于冗余。注意不要让对话中讨论过的压缩/提炼原则影响导出行为——导出是记录，不是总结。

### 3. 组织输出格式

文件头部：

````markdown
# {会话主题}

> {一句话概述本次会话的背景、目的和主要结论}

- 导出时间：YYYY-MM-DD HH:MM（使用用户本地时区）

> ⚠️ 注意：本次导出可能因上下文压缩而缺少早期对话内容。
<!-- 如果对话缺少自然开头（如首轮就是深入讨论、缺少背景铺垫），视为可能截断，添加上面这行。拿不准时加上。 -->
````

按对话轮次逐轮呈现：

````markdown
---

**💬 用户：**

{用户消息}

**🤖 AI：**

{AI 回复，保留原有 Markdown 格式}

---

**💬 用户：**

{用户消息}

**🤖 AI：**

{AI 回复}
````

格式要点：

- 标题从对话内容自动提炼
- 用 `**💬 用户：**` / `**🤖 AI：**` 加粗+emoji 标记角色，避免与回复中的 Markdown 标题冲突

### 4. 写入文件并报告

保存后告知用户：文件路径、导出轮次数。如果检测到可能的截断，提示用户早期内容可能丢失。
