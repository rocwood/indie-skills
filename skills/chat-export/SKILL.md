# Chat Export

将当前 AI 对话导出为 Markdown 文件。通过整理 LLM 上下文窗口中的内容实现，不依赖平台存储格式，适用于支持对话上下文的 Agent 工具。

**已知限制**：若会话经历上下文压缩或超出窗口，早期内容可能丢失。长会话建议中途分段导出。

## 步骤

### 1. 确定输出路径

按优先级选择，**不要反问用户**，直接决定：

1. 用户明确指定的路径
2. 从工作区上下文推断合理位置（如笔记库的日记目录、项目的 docs 目录等）
3. 工作区根目录

文件名：`YYYYMMDD-{主题关键词}.md`，日期取会话开始日期，主题从对话内容自动提炼（2-10 个字，仅使用中文、英文、数字和连字符）。
如果工作区有明确的文件命名惯例，遵循该惯例。

### 2. 回顾上下文，提取对话

**⚠️ 导出是记录，不是总结。** 不管当前对话的主题是什么（哪怕正在讨论"如何精简文字"），导出时不受对话内容的影响。你的任务是忠实还原对话原貌。

扫描上下文窗口，按时间顺序逐轮提取。**不包含"执行导出"这一轮本身。**

**过滤**（只过滤机械性噪音，不确定时不过滤）：

- 系统注入内容（打开的文件列表、附加文件全文、workspace rules、skills、用户环境元数据等）
- 工具调用的参数细节和原始返回值
- AI 的机械性内部过程（thinking / chain-of-thought 标签内容、TODO 状态更新操作等，但 TODO 列表本身若体现了有价值的任务拆解则保留）
- 工具调用替换为括号标注，一句话以内：`（读取了 SKILL.md）`、`（修改了 SKILL.md）`。这是元信息标记，不是 AI 的回复——和 AI 原文各司其职，不要用它替代或重述原文中已有的说明

AI 写给用户的文字（分析、论证、衔接、提问）都是回复内容，不过滤，无论出现在工具调用之前、之间还是之后。

**保留规则**：

- 用户消息原样保留。如果用户消息通过 `@文件名` 引用了文件，在消息后用括号注明实际路径，如 `（引用了 path/to/file.md）`
- AI 回复遵循两条原则：
  - **原文照搬**：逐字逐句保留 AI 写的文字，不改写措辞（哪怕语义相同）、不转换人称、不替换口吻。"好问题。让我对照这次总结来诊断。"就是原文，不要变成"读取了文件，对照诊断出问题："
  - **完整不删**：保留回复的全部结构和篇幅。多行展开的论述不压缩成一行结论；过程性衔接语（"让我验证一下""来通读一下最终版本"）不当作内部过程过滤；尾部的对照分析不因为紧跟工具操作就丢弃。原文用几行，导出就用几行
- 保留原有 Markdown 格式（加粗、斜体、列表层级）

**兜底原则**：拿不准是否该保留时，保留。信息丢失的代价远大于冗余。

### 3. 组织输出格式

文件头部：

````markdown
# {会话主题}

> {一句话概述本次会话的背景、目的和主要结论}

- 会话日期：YYYY-MM-DD（取会话开始日期，不是导出执行时间）

> ⚠️ 注意：本次导出可能因上下文压缩而缺少早期对话内容。
<!-- 如果对话缺少自然开头（如首轮就是深入讨论、缺少背景铺垫），视为可能截断，添加上面这行。拿不准时加上。 -->
````

按对话轮次逐轮呈现：

````markdown
---

**💬用户：**

{用户消息}

**🤖AI：**

{AI 回复，保留原有 Markdown 格式}

---

**💬用户：**

{用户消息}

**🤖AI：**

{AI 回复}
````

格式要点：

- 标题从对话内容自动提炼
- 用 `**💬用户：**` / `**🤖AI：**` 加粗+emoji 标记角色，避免与回复中的 Markdown 标题冲突

### 4. 写入文件并报告

保存后告知用户：文件路径、导出轮次数。如果检测到可能的截断，提示用户早期内容可能丢失。
