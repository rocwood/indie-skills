# 认知提炼

从会议纪要、讨论文档中提取已确认结论，维护到对应领域的 `__认知__.md` 文件中。
流程分为**提炼**和**审核**两个阶段——提炼完成后必须执行审核（见下方"审核"部分）。

## 三层结构定义

`__认知__.md` 采用三层结构：

- **内核（红线）**：定义"这是什么产品"——即产品的核心体验。一个产品真正只有一个唯一内核，能被一句话定义，并成为所有设计、所有判断的共同依据。内核不能轻易改；改了等于方向级变更，需团队全员共识。
- **体验表达（设计规则）**：体验表达要围绕内核服务。核心体验不等于主题包装，不等于方案，不等于局部卖点——体验表达是内核如何被用户感知到的设计原则。改动需有意识地追问"是否还在服务内核"，因为改一条会连带影响其他设计。
- **方案（当前选择）**：方案必须围绕内核服务，且方案是可替换的。如果不做这个区分，团队很容易把"方案"误当"核心"。

另有 **❓ 未决** 区——已提出但尚无共识的问题，不作为约束引用。

## 核心原则

- **先写后审**：直接更新认知文件，审核完成后向用户展示变更清单供 review。用户可随时回退或调整。
- **结论而非讨论**：只提取"已确认的判断"。判断标准：这句话能否在未来的方案评审中被引用为约束或依据？能 = 入库，不能 = 过滤。
- **一句话可引用**：每条认知的粒度是一个完整判断，用句号收束。不能是模糊的方向，必须是可检验的约束。
- **忠于来源**：不添加源文件中没有的判断。提炼可以精简表述，但不能改变含义或升级确定性（"倾向于 X"不能写成"确定 X"）。

## 认知文件模板

`__认知__.md` 放在对应领域目录下。定位规则：
- 源文件路径中有明确的项目/小组目录 → 认知文件在同目录
- 源文件分散在多个目录 → 认知文件在它们的最近公共父目录
- 不确定时询问用户

新建时按此模板生成：

```markdown
# [领域名] - 认知与共识

> 最后更新：YYYY-MM-DD

## 内核（红线）

- 最上位结论 [[源文件名|简短标识]]
  - 派生红线A [[源文件名|简短标识]]
  - 派生红线B ∵ 为什么这么定 [[源文件名|简短标识]]
    - 更细一层的约束 [[源文件名|简短标识]]
- 另一条顶层红线 [[源文件名|简短标识]]

## 体验表达（设计规则）

### [子领域A]

- 结论内容 → 服务于「内核条目关键词」 [[源文件名|简短标识]]
  - 派生/补充说明 [[源文件名|简短标识]]

### [子领域B]

- 结论内容 [[源文件名|简短标识]]

## 方案（当前选择，可替换）

### [子领域A]

- 结论内容 → 服务于「体验表达条目关键词」 [[源文件名|简短标识]]
  - 具体参数或细节 [[源文件名|简短标识]]

## ❓ 未决（尚无共识）

- 问题描述 [[首次提出的源文件名|简短标识]]
```

说明：
- 每个层级都可以用三级标题按子领域组织（内核条目少时可省略子标题）
- 列表项支持缩进子项表示派生关系或补充细节
- `→ 服务于「关键词」` 标注跨层级派生关系（体验表达服务于哪条内核，方案服务于哪条体验表达），关系明显时可省略
- 内核和体验表达条目可追加 `∵ ...` 标注意图（为什么这么定、成立前提、关键假设等），便于未来检核是否仍成立。方案层通常不需要
- 子领域标题按内容自然归类，无需预设

## 操作流程

### 提炼

用户提供源文件（会议纪要、讨论文档等）后：

**1. 读取**

- 按定位规则读取对应的 `__认知__.md`（不存在则视为空）
- 按时间顺序读取源文件（多个文件时先早后晚，后面的可能修正前面的）

**2. 提取**

扫描以下信号词识别已确认结论：
- 明确共识："结论是"、"确定"、"一致认为"、"明确了"、"核心结论"列表中的条目
- 否决/红线："坚决不要"、"明确否决"、"不做"、"去掉"
- 修正："修正为"、"之前说的 X 不对"、"纠偏"、"取代"
- 注意：出现在"未决问题"章节的条目默认归入未决，除非后文已有明确结论

**3. 分类与比对**

- 对每条结论进行层级判断（见下方"层级判断"）
- 与已有认知比对（已有为空则全部为新增）：
  - **新增**：之前没有的结论
  - **修正**：推翻或细化了已有条目
  - **强化**：为已有条目增加子项（合并处理，不单独列出）
  - **已有**：重复或已涵盖，跳过
- 同一话题有前后修正时，只保留最新版本

**4. 写入**

- 新增条目插入对应层级的合适位置（同子领域内追加在末尾）
- 修正条目直接替换原文
- 更新文件头"最后更新"日期
- 如需新增子领域标题，放在该层级已有子领域之后
- 文件不存在时按模板新建

### 审核

提炼完成后，进入审核阶段。审核是独立于提炼的第二阶段——切换到"审核人"角色，假设认知文件是别人写的，你负责找问题。

**审核过程必须可见**：使用当前环境提供的任务管理工具（如 TodoWrite、update_plan、TaskCreate 等）将以下步骤注册为独立任务，并逐项更新状态。

按以下步骤逐项执行：

1. **内容准确性（第一轮）**：重新通读源文件，逐条检查已写入的认知——是否真的是"确认"而非"探索"？粒度是否合适（太粗则拆为多条，太细则抽象为上层原则）？层级是否准确（用三个判断问题再过一遍）？是否和已有条目重复或矛盾？wikilink 源文件名是否正确？发现问题立即修正
2. **内容准确性（第二轮）**：针对第一轮修正后的认知文件，再通读一遍源文件，确认没有因修正而引入新的遗漏或错误
3. **展示变更清单**：向用户展示最终变更结果（见下方模板），拿不准的条目标注 `(?)` 提示用户注意

**变更清单模板：**

```
已更新 __认知__.md，本次变更：

📥 新增：
- [内核] xxx [[源文件|标识]]
- [体验表达/子领域] yyy [[源文件|标识]]
- [方案/子领域] zzz [[源文件|标识]]
- [未决] aaa

✏️ 修正：
- 原：bbb → 改为：bbb' [[新源文件|标识]]

🗑️ 移除/降级：
- ccc → 降级为 XX（原因：...）

如有需要调整的，直接说。
```

## 判断标准

### 什么该入库

- 明确达成共识的判断——有收束表述、在"核心结论"中出现、或被决策者确认
- 被明确否决的方向——以红线的反面表述入库（"不做 X"而非"讨论过 X"）
- 对已有认知的修正——新结论取代旧结论

### 什么不该入库

- 行动项、任务分工、排期——属于项目管理
- 当前问题/体感反馈——"用户体验不好"是问题不是认知
- 工作方法论——有独立文件承载
- 过程讨论和分歧呈现——"A 提出...B 反驳..."
- 探索性想法——"可以考虑..."、"有想象空间"（放未决或不录入）
- 具体案例——案例是论据不是结论，只有从案例中提炼出的规则才入库

### 层级判断

三个自问，逐条过：

1. **内核**："如果推翻这条，我们还在做同一个产品吗？" → 是 = 内核
2. **体验表达**："这条被改了，需要连带审视其他设计吗？" → 是 = 体验表达
3. **方案**："换一种实现方式，上层完全不受影响？" → 是 = 方案

灰色地带的处理：拿不准时**宁可降级**（放体验表达而非内核，放方案而非体验表达），并在变更清单中标注 `(?)` 让用户裁决。

## 表述规范

- 每条用一个完整判断句，句号收束
- 不用"应该"、"建议"等软性措辞——认知文件里的都是已确认的
- 否定式约束用"不要/不做"开头，肯定式用名词或动词开头
- 派生关系用 `→ 服务于「...」` 在同一行末尾标注，不另起行
- wikilink 使用 `[[文件名（不含路径）|日期+关键词]]` 格式，如 `[[20260612-合体案例爽感要素讨论会-总结|6/12爽感讨论]]`
- 同一源文件被多条引用时，显示文本保持一致

### 要避免

- 把行动项伪装成认知（"下周要做 X" → 不入库）
- 把体感反馈当结论（"现在体验不好" → 不入库）
- 堆砌同义反复的条目——如果两条说的是同一件事，合并为一条
- 照搬源文件的长段落——必须压缩为一句话
- 升级确定性——源文件说"初步方向"就不能写成"已确认"

## 特殊情况处理

- **源文件无明确结论**：告知用户"这份材料中未识别到已确认结论，是否有我遗漏的？"不强行提取。
- **同一会议内自相矛盾**：以会议后半段的结论为准（通常前面被纠偏），在变更清单中标注两处出处供用户裁决。
- **结论的确定性模糊**："初步的方向感"、"倾向于"、"待验证"——降级为未决或方案层，在变更清单中标注 `(?)`。
- **大量新增（>8条）**：分子领域逐批展示，避免一次性信息过载。
- **用户要求回退**：按用户指示移除或修改对应条目，不追问原因。
