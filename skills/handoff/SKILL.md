# Handoff

将当前 AI 会话导出为 Markdown 格式的交接、分享文件，供另一个 Agent 接手后续工作。也可以作为分享材料，给没有参与会话的人阅读。导出时在忠实记录的基础上过滤干扰信息。

## 步骤

### 1. 确定输出路径

按优先级选择，**不要反问用户**，直接决定：

1. 用户明确指定的路径
2. 从工作区上下文推断合理位置（如笔记库的收集箱、项目的 docs 目录等）
3. 工作区根目录

文件名：`YYYYMMDD-{主题关键词}-handoff.md`，日期取会话开始日期，主题从会话内容自动提炼（2-10 个字，仅使用中文、英文、数字和连字符）。如果工作区有明确的文件命名规则，遵循该规则。

### 2. 回顾上下文，提取会话

**⚠️ 会话正文是记录，不是总结。** 忠实保留会话的实质内容，不改写、不压缩，也不受当前会话主题影响（哪怕正在讨论"如何精简文字"）。

扫描上下文窗口，按时间顺序逐轮提取。**不包含"执行导出"这一轮本身。**

**过滤**（只过滤机械性和无信息量的噪音；当不确定时不过滤）：

- 过滤系统注入内容（打开的文件列表、附加文件全文、workspace rules、skills、用户环境元数据等）
- 过滤 AI 的机械性内部过程（thinking / chain-of-thought 内部推理等）
- 工具调用只保留三类——创建 TODO/Task 任务清单、修改/写入文件、执行有副作用的命令（git 提交、运行脚本等）——用一句括号标注，格式为（动作 + `文件名/操作目标/命令`），如 （修改了 `SKILL.md`）、（执行了 `git commit`）；其余工具调用（读取、搜索、只读命令如 `ls/cat` 等）整个过滤掉
- 过滤过程性自言自语：AI 边做边汇报在读什么、下一步打算做什么——做完就失去价值的话，哪怕夹带零碎观察（如"文件挺长"、"从 626 行开始"），如 "Let me continue reading the rest."

**保留**：

- 用户消息原样保留；消息中的 `@文件名` 引用保留 `@` 前缀，路径统一为基于工作区的相对路径
- AI 回复：写给用户的文字（分析、论证、衔接、提问）都算回复内容，无论在工具调用之前、之间还是之后（过程性自言自语除外）。保留时遵循两条原则：
  - **原文照搬**：逐字逐句保留 AI 写的文字，不改写措辞（哪怕语义相同）、不转换人称、不替换口吻。"好问题。让我对照这次总结来诊断。"就是原文，不要变成"读取了文件，对照诊断出问题："
  - **完整不删**：保留回复的全部结构和篇幅。多行展开的论述不压缩成一行结论；尾部的对照分析不因为紧跟工具操作就丢弃。原文用几行，导出就用几行
- 保留原有 Markdown 格式（加粗、斜体、列表层级、标题）

**兜底**：拿不准是否该保留时，保留。信息丢失的代价大于冗余。

### 3. 输出格式

文件头部：

````markdown
# {会话主题}

> {一句话概述本次会话的背景、目的和主要产出}

- 会话日期：YYYY-MM-DD（取会话开始日期，不是导出执行时间）
- 会话轮次：N 轮

## 会话大纲（可选）

1. {一句短句概括这个阶段}（第 1 轮）
2. {一句短句概括这个阶段}（第 2-4 轮）
````

会话大纲按需生成：3 轮及以内省略大纲；4 轮及以上才加，相关轮次合并成阶段并标范围（如"第 2-4 轮"），并用短句概括内容，不要一轮一条。

按会话轮次逐轮呈现，每轮加 `## 第 N 轮` 标题，轮次之间用 `---` 分隔：

````markdown
---

## 第 1 轮

**💬用户：**

{用户消息}

**🤖AI：**

{AI 回复}

---

## 第 2 轮

**💬用户：**

{用户消息}

**🤖AI：**

{AI 回复}
````

格式要点：

- 用 `**💬用户：**` / `**🤖AI：**` 加粗+emoji 标记角色

### 4. 写入文件并报告

保存后告知用户：

- 文件路径、导出轮次数
- 若检测到上下文压缩（有"先前会话摘要"/压缩标记，或会话缺少自然开头这类辅助迹象），说明早期轮次可能已折叠或丢失、轮次编号未必对应真实起点；提示用户，并建议长会话中途分段导出
- 若内容涉及敏感信息（人名、项目名、文件路径等），提醒对外分享前留意脱敏
